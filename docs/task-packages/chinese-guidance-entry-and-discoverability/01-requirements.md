# Requirements

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## Goal
让中文维护者在首次进入 OpenHarness 时，不需要先读懂多个英文 skill，也能知道 task package 的五类核心文档分别该看哪里、该怎么开始写，以及当前 guidance 的正式入口在哪里。

本轮的单一成功指标是：一个看不懂英文 skill 的维护者，仍然能仅凭仓库中的正式入口和中文说明，找到五份 guidance 并理解自己在当前阶段该先看什么。

## Problem Statement
OpenHarness 最近已经补上了五份分文档 guidance，这意味着它不再只是“七个文件名 + 状态流”的空协议。但当前仍然存在一个明显缺口：真正有用的写作说明主要藏在 `references/` 和多个 skill 之间，而不是直接暴露给中文使用者。

当前问题具体体现在：

- 用户第一眼看到的仍然是英文 `SKILL.md`、英文 skill 名和较轻量的模板，而不是一个中文作者入口。
- `OH-033` 已经补齐 guidance 内容，但“guidance 在哪里、什么时候该看哪一份”仍然依赖用户理解 skill 路由。
- 模板虽然不再完全空白，但提示仍偏短，无法单独承担“打开即会写”的入口职责。
- bootstrap 当前能列 active task package，但还没有把“如果你是中文作者，现在该看哪份正式说明”直接暴露出来。
- 这会让外部观察者继续产生“这个项目是不是只有形式和文件结构”的合理怀疑，即便仓库内部其实已经有更实质的内容。

现在要做这件事，是因为仓库已经完成第一轮写作方法论补齐，如果不继续补入口和可见性，真实价值仍然会被协议外壳遮住，导致新增的方法论难以被普通维护者使用。

目标用户主要是：

- 看不懂英文 skill 的中文维护者和协作者。
- 首次创建或阅读 task package 的新使用者。
- 想判断 OpenHarness 是否真的有指导意义，而不是只是在堆协议的人。

核心场景是：用户从仓库入口进入，准备理解或创建一个 task package，需要迅速知道当前阶段应该看什么、五类文档分别由什么说明约束，以及如何避免把 task package 写成形式主义空壳。

## Required Outcomes
1. 明确 OpenHarness 当前对中文使用者最关键的可用性缺口，不把问题泛化成“全部英文都要重写”。
2. 定义一个稳定的中文作者入口，说明它应放在什么仓库表面，负责什么，不负责什么。
3. 明确 bootstrap、README、skill-hub、关键 stage skill 和模板之间的导流职责，避免 guidance 已存在却难以被发现。
4. 给出模板增强的最低方案，使五类文档的模板提示比现在更具体，但仍然保持轻量。
5. 给出后续实现要改哪些文件、先后顺序如何安排、用什么命令和场景验证效果。

补充 requirements gate 要点如下：

- 目标用户：中文维护者、新协作者、首次接触 task package 的作者。
- 核心场景：用户进入仓库后，需要在不依赖英文 skill 深读的前提下找到 guidance 与写作入口。
- 单一成功指标：中文使用者能靠正式入口定位五份 guidance，并知道当前阶段先看哪份。
- 成本上限：本轮不做全仓库全面汉化，不重写所有历史 task package，不引入新的固定 task package 文件。
- 验收标准：形成一套清晰的入口设计，后续实现后可以显著降低“guidance 已存在但看不见”的问题。
- 反例：如果只是新增另一份长文档，却没有把入口挂到用户真正会看到的位置，就不算完成。

## Non-Goals
- 本轮不把所有 skill 正文都翻译成中文。
- 本轮不重做 task package 七文件协议，也不新增平行文件体系。
- 本轮不把模板扩成完整手册。
- 本轮不把 `check-tasks` 升级成自然语言质量评分器。
- 本轮不一次性修订所有 archived task package 的历史内容。

## Constraints
- 必须保持当前 task package 文件结构、状态流和 guidance 分层不变，除非单独开包升级协议。
- 必须尊重 `OH-033` 已经确立的分层：skill 讲动作，guidance 讲写法，template 讲最短提示，validator 讲最低诚实性检查。
- 新增入口必须优先解决“发现路径”问题，而不是制造新的事实源分散。
- 仓库正文仍应以中文为主，但章节标题、状态值、YAML 键名、文件名和路径继续保持英文。
- 如果某个改动要求大量改 CLI 行为或新增复杂交互，应拆成后续 task package，不在本轮隐式扩张。
