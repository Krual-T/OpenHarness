# Overview Design

## System Boundary
本轮设计只覆盖 `openharness` 的方法论层，不覆盖代码实现层。它关注的是：

- 现有阶段骨架如何承载更强的方法论
- 多视角角色应在什么阶段介入
- 主代理如何组织和收敛多智能体输入
- 哪些能力是可选辅助，而不是默认基线

本轮不覆盖：

- 浏览器安装、页面渲染、可视化工具接入
- 具体子智能体提示词实现
- 仓库技能文件的代码改造

## Proposed Structure
采用“两层模型”，而不是新增一套平行工作流。

第一层是阶段骨架，直接复用 `openharness` 现有主线：

1. `brainstorming`
2. `exploring-solution-space` / 总体设计前探索
3. `02-overview-design.md`
4. `03-detailed-design.md`
5. `05-verification.md`

第二层是在每个阶段注入角色视角：

- 需求阶段注入产品视角、CEO 视角
- 总体设计阶段注入架构视角、产品视角
- 详细设计阶段注入架构视角、测试视角
- 验证阶段注入 review 视角、风险视角

角色不是新的主流程节点，而是阶段内的挑战机制。主代理始终负责推进当前阶段，辅助角色只负责在限定边界内提出质疑和补充。

为了防止角色注入流于评论，还需要补一层“阶段门禁”：

- `brainstorming` 结束前必须明确目标用户与核心场景、唯一成功指标、明确不做清单、成本上限，并给出可测试的验收标准与至少一个反例。
- `02-overview-design.md` 结束前必须明确关键约束、接口边界、关键失败模式、降级或回滚思路。
- `03-detailed-design.md` 结束前必须明确测试策略矩阵、可观测性要求、验证证据类型、迁移与回退边界。
- `05-verification.md` 结束前必须明确需求到验证的追踪关系、剩余风险清单和接受理由。

这个结构的关键约束是：

- 阶段是主轴，角色是注入物，不允许反过来由角色主导全流程
- 主代理先产出草案，再让辅助角色做定向挑战，最后由主代理收敛
- 浏览器、视觉材料、界面工具只在确有视觉问题时作为可选载体使用，不进入默认协议
- 阶段之间的推进要受门禁约束，不允许靠更长的描述掩盖关键决策未完成

## Key Flows
主流程如下：

1. 用户提出一个目标，主代理先判断当前处于哪个阶段。
2. 主代理产出该阶段的初稿，例如需求草案、总体方案草案或详细设计草案。
3. 系统根据阶段自动选择应注入的视角角色，而不是任意拉一组代理同时发言。
4. 各角色只围绕本阶段允许挑战的问题给出反馈。
5. 每条关键挑战都必须被主代理标注处置结果：接受并落为约束、拒绝并给出理由与替代、延期并写清触发条件与最晚落点。
6. 主代理汇总冲突、做取舍，并把结论写回阶段文档。
7. 当当前阶段已具备可执行结论并满足阶段门禁时，再推进到下一阶段。

信息流不是“多代理同时产出多个版本”，而是“单一主稿 + 定向挑战 + 收敛修订”。

推荐的收敛优先级如下：

1. 明确违反需求边界或非目标的意见不进入实现侧约束。
2. 明确影响可验证性、可回滚性、可观测性的意见优先级高于表层体验优化。
3. 明确影响用户价值或投入产出比的意见优先级高于风格偏好。
4. 无法在当前阶段闭合的争议不得被伪装为已解决，必须被记录为延期决策或回退理由。

## Trade-offs
该方案的收益：

- 保留 `openharness` 已有阶段骨架，避免另起炉灶
- 能把 `gstack` 的价值落在方法论和协作协议上，而不是外部工具上
- 让多智能体协作从“多人同时聊天”变成可控的阶段性挑战机制

代价与限制：

- 阶段内的角色注入规则需要额外设计，不能只靠模糊直觉
- 主代理的收敛责任更重，需要明确决策依据
- 如果后续实现时角色职责定义不清，仍可能出现重复发言和边界混乱
- 如果没有门禁、否决条件和时间盒，流程会变成“写得更长、决策更慢”

没有选择“按角色组织系统”的原因是，那样会削弱阶段主线，使流程更容易碎片化。

没有选择“复制浏览器或界面能力”的原因是，那属于具体交互媒介，不是跨项目稳定成立的方法论。

## Overview Reflection
我显式挑战了三个备选方向：

- 把 `gstack` 价值理解为浏览器或前端交互能力。这个方向不成立，因为它无法覆盖后端、基础设施和纯文档任务。
- 重新发明一套平行于 `openharness` 的阶段模型。这个方向不成立，因为当前缺口不是阶段数量不够，而是阶段内缺少多视角注入协议。
- 让多个角色全程常驻发言。这个方向不成立，因为它会制造持续噪声，削弱主路径和收敛责任。

结论是：保留现有阶段骨架，补充横向角色注入机制，是最小且最稳的路径。

在本轮的后续探索里，我补做了有边界的子智能体评审，分别从产品/CEO 和架构/测试两个方向挑战方案。两个评审都给出 `approve_with_changes`，核心意见一致：

- 必须补阶段硬性产出，不能只写“注入什么角色”
- 必须补挑战闭环，避免角色只提意见不形成约束
- 必须补主代理收敛判据，避免靠写作能力压过异议
- 必须限制 CEO 与风险视角的边界，避免流程过重

这些增量已经吸收到当前设计中，因此本轮 overview 结论比首稿更稳。
