# Requirements

## Goal
把 `gstack` 中真正有价值的工作方法提炼出来，并转译为 `openharness` 可直接采用的阶段化方法论与多视角协作框架。

本轮目标是形成清晰设计，而不是实现新工具链，也不是把浏览器、界面或某个具体运行时当成默认依赖。

## Problem Statement
当前诉求不是“安装更多能力”，而是避免把能力理解错位。若把 `gstack` 简化成浏览器、前端演示或某种固定工具组合，会带来几个问题：

- 把可选载体误当成核心能力，导致后端项目、基础设施项目或纯文档任务被迫背上无关依赖。
- `openharness` 虽然已经有 `brainstorming`、`overview design`、`detailed design`、`verification` 这类阶段骨架，但每个阶段缺少更明确的多视角挑战协议。
- 现有阶段更多解决“流程顺序”和“文档落点”，还没有充分回答“谁在什么阶段介入、挑战什么问题、由谁收敛分歧”。
- 如果不先把方法论和工具形态拆开，后续无论实现多少辅助能力，都容易继续偏离目标。

## Required Outcomes
1. 明确区分“应吸收的方法论能力”和“只在特定场景下可选的工具能力”。
2. 把目标定义为“按阶段组织、按角色注入”的框架，而不是平行于 `openharness` 的另一套流程。
3. 为 `brainstorming`、总体设计、详细设计、验证等阶段分别定义应注入的角色视角。
4. 为每个角色定义职责边界，说明它主要挑战哪些问题，不应越界做什么。
5. 明确主代理与辅助代理的协作方式，避免多智能体讨论失控。
6. 形成可继续推进到提示词、技能设计或协作协议实现的文档基础。

## Non-Goals
- 不在本轮实现浏览器能力、前端界面或具体 agent 运行时。
- 不在本轮修改任何仓库代码或技能实现。
- 不把所有角色都做成常驻主视角；角色只在适合的阶段注入。
- 不复制一套平行于 `openharness` 的新阶段模型。

## Constraints
- 必须兼容现有 `openharness` 阶段骨架，而不是推翻重来。
- 任务包正文保持中文，章节标题、状态值、命令、文件名与路径保持英文。
- 本轮只进行讨论和文档沉淀，不进入实现状态。
- 浏览器能力只能被视为可选视觉辅助，不能被抽象成所有项目的默认依赖。
