# Overview Design

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## System Boundary
这轮 overview 覆盖三类表面：

- OpenHarness 当前仓库的活跃协议与实现表面：
  - `AGENTS.md`
  - `README.md`
  - `skills/`
  - `openharness_cli/`
  - `tests/`
- OpenHarness 的历史使用案例表面：
  - `docs/archived/task-packages/` 中能直接反映验证闭环、状态语义、stage gate、协议去重和维护成本的归档 package
- 外部正式对照表面：
  - `superpowers`
  - `OpenSpec`
  - `spec-kit`
  - 以及与它们直接相关的官方 README / 文档结论

这轮不覆盖的内容：

- 不直接修改 OpenHarness 协议、CLI、模板或测试。
- 不把所有开源 agent workflow 项目都纳入生态综述，只保留能直接形成结构对照的少数对象。
- 不在这轮先决定具体文件怎么删、怎么迁移；那属于下一轮重构 task package。
- 不把“是否兼容旧结构”当成既定前提。本轮只负责判断哪些旧结构值得被下一轮继续保留。

## Proposed Structure
推荐采用“三层证据、两步收敛、一个输出边界”的结构。

三层证据：

- 第一层是当前仓库事实：
  - 看 OpenHarness 自己现在怎样定义入口协议、task package、状态流、验证和归档。
  - 这里的重点不是重复 README 自述，而是检查这些规则是否真的落进 CLI、校验器和测试。
- 第二层是归档案例：
  - 把归档 task package 当成真实使用案例，重点看它们是否记录了假完成风险、状态语义修复、reflection 带来的真实决策变化、以及协议维护成本。
  - 本轮优先使用这些包：`workflow-transition-and-verification-artifacts`、`task-package-completion-semantics`、`workflow-stage-visibility-and-task-intake`、`workflow-redesign`、`python-verification-maturity`、`maintenance-and-entropy-reduction`、`skill-protocol-deduplication`。
- 第三层是外部对照：
  - `superpowers` 作为“代理执行能力层”对照。
  - `OpenSpec`、`spec-kit` 作为“规格驱动工件层”对照。
  - OpenAI / Anthropic 官方材料作为“理念与生态层”背景，不把它们误当成和 OpenHarness 一一对应的成套仓库协议。

两步收敛：

- 第一步先回答两个核心判断：
  - OpenHarness 是否真实减少假完成。
  - `reflection` / `stage gate` 是否真实改变设计和验证决策。
- 第二步再把当前结构拆成三类：
  - 应保留的必要闭环
  - 应削减的流程负担
  - 应在下一轮直接重组且不预设兼容性的部分

一个输出边界：

- 本轮的最终输出不是“再补一套抽象评价原则”，而是下一轮重构边界。
- 这个边界必须明确指出哪些结构是 OpenHarness 的核心价值，哪些结构已经接近流程负担，哪些部分下一轮应允许直接删改而不是继续兼容包袱。

## Key Flows
主流程分成四步：

1. 先建立当前系统模型。
   - 从现有 `AGENTS.md`、skills、CLI、validation、lifecycle 和测试里确认 OpenHarness 现在真正实现了什么。
   - 关键失败信号是：README 说有某种能力，但代码和测试里并没有对应的硬约束。
2. 再用归档 package 抽取内部案例。
   - 重点看哪些历史包真正修复了假完成、状态失真、协议重复、维护漂移等问题。
   - 关键失败信号是：归档包里只有口号式 reflection 或 verification 记述，却无法证明这些步骤真的改变了结构或验证策略。
3. 再和外部项目做分层对照。
   - 判断 OpenHarness 和 `superpowers`、`OpenSpec`、`spec-kit` 分别在什么层面可比，避免把不同层级的东西硬拉成同类比较。
   - 关键失败信号是：把“有 skills”“有文档”“有 eval”这类表面相似错当成结构等价。
4. 最后输出下一轮重构边界。
   - 把现有结构分成保留、削减、重组三类，并明确哪些部分不应被兼容性约束绑住。
   - 关键失败信号是：结论仍然停留在“有优点也有风险”，却没有落成下一轮到底改什么。

## Stage Gates
- 必须至少用一组归档 package 证明“假完成风险”和“协议维护成本”不是抽象担忧，而是仓库里已经真实出现过、并被某些任务显式修复过的问题。
- 必须明确 `superpowers`、`OpenSpec`、`spec-kit` 分别作为哪一层对照对象，不能继续把它们混成一个笼统的“外部最佳实践”。
- 必须给出 OpenHarness 当前结构的初步分类：
  - 哪些是必要闭环
  - 哪些是可接受成本
  - 哪些已接近形式主义或重复协议
- 必须明确至少一个失败时的收缩方向：
  - 如果某些判断缺少证据，本轮结论应收缩为“暂不下判断”，而不是补更多抽象原则。
- 必须把下一轮重构边界写到足够具体，能够单独开包推进，而不是再回到需求争论。

## Trade-offs
收益：

- 直接使用归档 package 作为内部案例，能避免“我记得好像有过”的口述不确定性。
- 把外部项目按层次比较，而不是按热度比较，能更快看清 OpenHarness 的真实定位。
- 把最终输出收敛到“下一轮重构边界”，能防止本轮再次沦为一份长篇项目评论。

代价：

- 归档 package 本身带有历史语义，某些旧包仍保留早期协议痕迹，需要区分“历史事实”和“当前最佳实践”。
- 外部对照材料质量并不完全一致，尤其 `OpenSpec` 和 `spec-kit` 系列更像工件规范层，不能直接替代仓库协议层比较。
- 不在本轮直接动代码，意味着有些问题只能先界定边界，不能立即验证重构收益。

不选的方案：

- 不选“继续做一份更完整的生态调研”，因为这会扩大资料面，却不一定更接近下一轮重构边界。
- 不选“只根据当前仓库自评做结论”，因为这样容易把内部叙事当成外部竞争力。
- 不选“先假设必须兼容现有结构再设计下一轮”，因为这会提前锁死最重要的重组空间。

## Overview Reflection
我先挑战了“没有用户现成案例，这轮是不是就无法继续”。结论是否定的。OpenHarness 的归档 task package 本身就是仓库内历史事实源，只要挑选得当，足以承担第一轮内部案例职责，因此接受“先用归档包代替口述案例”的路径。

我再挑战了“这轮是否应该先写一份更通用的评分框架，再决定看哪些包”。结论是否定。因为当前任务的目标不是发表通用方法论，而是为 OpenHarness 下一轮重构定边界，所以评估必须直接围绕仓库表面和具体历史包展开。这个挑战被拒绝。

我也挑战了“是否应该把兼容性继续作为默认约束”。结论是否定。多个归档包已经显示 OpenHarness 在关键语义收敛时，往往是通过重命名、去重、收紧状态语义、明确归档语义来前进的，而不是永远保留双重解释。因此本轮接受“下一轮设计不预设兼容保留”这一约束。

延期的问题是：是否需要在这轮就把下一轮重构拆成多个子方向。当前先延期，触发条件是 overview 结束后发现“保留 / 削减 / 重组”三类内容内部仍然跨度过大，单一重构包无法承载。
