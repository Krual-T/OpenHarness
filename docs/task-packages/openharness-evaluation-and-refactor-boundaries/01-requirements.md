# Requirements

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## Goal
为 OpenHarness 建立一份可以直接指导下一轮重构的客观评估 task package，先基于仓库内事实、真实使用案例和外部对照对象判断这套 harness 是否真的降低了假完成与协作失真，再据此给出下一轮重构应当触及什么、明确不该为兼容性保留什么。

单一成功指标是：评估结论足够具体，能够把下一轮工作收敛成一个边界明确的重构任务，而不是继续停留在“也许有价值，也许太重”的泛讨论。

## Problem Statement
OpenHarness 现在已经不只是一个理念说明仓库，而是拥有 `AGENTS.md`、task package、skills、CLI、校验器、验证写回和归档协议的一套实际工作流。问题在于，这些结构虽然越来越完整，但还没有经过一轮足够聚焦的反身评估来回答几个高价值问题：

- 这套 harness 是否真的减少了 agent 的假完成，而不是只让完成报告更长。
- `reflection`、`stage gate`、`challenge closure` 这些机制是否真实改变了设计或验证决策，而不是变成必填段落。
- 当前协议层、guidance、模板、测试和历史 task package 的同步面是否已经开始接近维护负担上限。
- 和外部公开项目相比，OpenHarness 的独特价值到底在什么地方，哪些结构是必要闭环，哪些结构已经过重。

目标用户首先是 OpenHarness 的维护者和高频使用者，其次是未来会接手这套仓库协议的人。核心场景不是“介绍项目是什么”，而是“在准备继续演进 OpenHarness 时，需要先判断哪里是真有效，哪里已经在滑向形式主义”。之所以现在做，是因为仓库已经积累了足够多的 task package、skills 和 CLI 约束，再晚做，后续重构很容易继续围绕现有结构做局部加法，错过主动减重和重组主路径的窗口。

这轮任务还带有一个明确约束：后续设计必须允许得出“不做兼容保留”的结论。也就是说，本轮不是为了证明现有协议都该被继承，而是为了给下一轮重构提供诚实边界。

## Required Outcomes
1. 形成一份基于仓库事实的 OpenHarness 评估框架，明确优先判断“是否减少假完成”“`reflection` / `stage gate` 是否有真实收益”这两个核心问题。
   `acceptance criteria`: 评估框架能够直接映射到 OpenHarness 当前仓库的具体表面，而不是只给通用方法论。
2. 把 `superpowers`、`OpenSpec`、`spec-kit` 作为正式对照对象，说明 OpenHarness 与它们分别在哪个层面可比、在哪个层面不可比。
   `acceptance criteria`: 对照结论以仓库、README、官方文档或其他可复查资料为依据，而不是只靠印象。
3. 纳入真实使用案例或至少归档 task package 级别的内部案例，判断 OpenHarness 的流程设计是否在真实任务中提前暴露风险、减少交接损耗或避免假完成。
   `acceptance criteria`: 至少能从现有 task package、验证记录、测试或历史结构中提取出可复查证据，而不是只做理念讨论。
4. 给出下一轮重构的明确边界，说明哪些部分值得保留、哪些部分应被削减、重组或直接替换，并明确本轮不预设兼容性保留。
   `acceptance criteria`: 后续可以据此单独开一个重构 task package，而不需要重新讨论“这轮到底要改什么”。
5. 产出一个新的 task package，并把后续探索、设计、验证入口全部安放在这个包内。
   `acceptance criteria`: 本包在本轮结束时至少应达到设计可继续推进的状态，而不是只有聊天中的临时结论。

## Non-Goals
- 本轮不直接修改 OpenHarness 协议、状态流、CLI 行为、模板或测试。
- 本轮不为了照顾历史包或现有实现而预先承诺兼容性；兼容性是否保留，应当是下一轮重构基于本轮评估再决定的事。
- 本轮不把所有外部项目做成大而全的生态调研；只围绕 `superpowers`、`OpenSpec`、`spec-kit` 这些直接对照对象收敛。
- 本轮不把“项目是否有理念价值”和“下一轮具体改哪里”混成一个长篇评论；必须落成可执行边界。
- `counterexample`: 如果只是继续补一篇 README 风格的项目自我介绍，或者继续往现有协议上叠加更多 guidance，却不回答“哪一层该被删、哪一层真有收益”，那不属于本任务包的完成结果。

## Constraints
- 本轮必须遵守现有 task package 协议，把主要结论写回本包，而不是停留在聊天中。
- 本轮的正式对照对象固定包含 `superpowers`、`OpenSpec`、`spec-kit`，不能被临时替换成更宽泛但更空的“社区趋势”。
- 评估必须优先使用仓库内可复查证据、真实使用案例、官方 README 或官方文档；如果某个外部项目证据不足，应明确说明不足，而不是硬下结论。
- 本轮先分析和设计，不做实现；如果探索过程中发现必须改协议才能继续，那也应被记录为下一轮边界，而不是在本轮直接动手。
- 兼容性不是本轮的默认约束。下一轮设计时必须允许得出“直接移除、重组或放弃旧结构”的方案，因此本轮不能把“保持向后兼容”写成先验要求。
- `cost cap`: 本轮只做到需求、探索、总体结论和下一轮重构边界，不把所有问题一次性解决，更不同时启动协议重构实现。
