# Requirements

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## Goal
为 OpenHarness 补上一套面向作者的 task package 文档写作方法论，让维护者不再只能靠英文 skill、零散模板提示和历史样例去猜每类文档该写什么、写到什么程度，以及这些指导应该放在哪里。

## Problem Statement
当前仓库已经有 task package 结构、状态流和最低语义校验，但仍缺少一层真正面向写作者的清晰指导。

已有现状如下：

- `SKILL.md` 更擅长描述阶段动作，例如先做什么探索、需要从哪些角度反思、什么时候进入下一阶段。
- 模板提供了每个章节的一两句提示，但深度只够提醒主题，不足以教会普通使用者如何写出合格文档。
- `check-tasks` 能拒绝明显空壳或占位内容，却不会告诉作者“系统边界到底应该怎么界定”“详细设计里的接口和迁移说明需要写到什么粒度”。
- 归档 task package 里有一些可参考样例，但它们分散在历史材料中，阅读成本高，而且对看不懂英文的维护者不友好。

这会造成几类具体问题：

- 仓库看起来已经有完整协议，实际却仍然依赖作者个人经验，容易退化成“按标题填空”的形式主义。
- 新协作者无法快速分辨哪些内容属于阶段方法，哪些属于文档写法，哪些只是某个历史任务的特例。
- 当用户明确提出“我看不懂英文，希望客观判断这个项目是否真的有用”时，仓库目前拿不出一份面向普通维护者的中文写作说明书。
- OpenHarness 可能继续在 skill、模板、样例和测试之间分散解释，导致维护成本持续上升。

这轮任务的背景还包括一个明确用户诉求：用户不仅要判断 OpenHarness 是否只是形式主义，还要推动它往“可教学、可执行、可审查的中文任务包方法论”方向演进。因此需求中必须把这种背景和真实动机写明，而不是只写抽象的“补文档”。

## Required Outcomes
1. 明确当前 OpenHarness 在 task package 文档写作指导上的缺口，并把用户背景、真实痛点和为什么现在要做写清楚。
2. 明确 `SKILL.md`、模板、`references/` 的职责边界，避免把阶段动作指导和文档写法指导继续混在一起。
3. 参考 OpenSpec 的做法，给出 OpenHarness 应该借鉴什么、不该照抄什么的结论。
4. 设计一份适合 OpenHarness 的写作方法论文档结构，至少覆盖 `01-requirements.md`、`02-overview-design.md`、`03-detailed-design.md`、`04-verification.md`、`05-evidence.md`。
5. 把后续实施路径具体化到可以直接进入执行，例如先新增哪类 reference、模板要改多少、skill 要不要只补链接与职责说明。

补充的 requirements gate 要点如下：

- 目标用户：主要是 OpenHarness 的维护者和协作者，尤其是看不懂英文或不熟悉设计写作方法的人。
- 核心场景：用户准备创建或阅读 task package，需要快速知道每个文档该写什么、不该写什么、写到何种粒度才算进入下一阶段。
- 单一成功指标：维护者在不依赖历史样例猜测的情况下，也能仅凭仓库内正式指导理解五类核心文档的写作边界。
- 成本上限：本轮只收敛方法论落点与内容结构，不把所有 skill 全量重写为中文，也不同时发明更重的 schema 或评分系统。
- 验收标准：形成清晰设计，能支持后续落地一份正式写作指南，并让模板与 skill 只承担各自应有的最小职责。
- 反例：如果本轮只是把更多说明散落塞进各个 `SKILL.md`，或者只是给模板多加几句口号，而没有形成统一写作指南，则不算完成。

## Success Conditions
- 仓库对“阶段怎么推进”和“文档怎么撰写”有稳定分层，而不是继续互相污染。
- 后续维护者能够明确知道“系统边界”“主流程”“验证路径”“残余风险”等章节分别要回答哪些问题。
- OpenHarness 能拿出一份比当前模板更具体、但又不至于重到 OpenSpec 那种完整规格系统的写作方法论方案。
- 本轮设计结论能够自然衔接到后续实现，不需要再重新开一轮需求澄清。

## Non-Goals
- 本轮不直接重写所有现有 skill 正文。
- 本轮不一次性修改所有 archived task package 去适配新的写作指南。
- 本轮不引入新的状态流、额外 task package 文件或另一套平行协议。
- 本轮不把 `check-tasks` 升级成质量评分器或自然语言审稿器。
- 本轮不照搬 OpenSpec 的 `proposal/design/tasks/specs` 文件布局。

## Constraints
- 必须保持当前 task package 文件结构与状态流稳定，除非后续单独开包讨论协议升级。
- 必须尊重当前仓库分层：`using-openharness` 是入口协议层，child skills 负责阶段动作，模板是脚手架，校验器是最低诚实性检查。
- 方案需要兼顾中文可读性与英文结构锚点稳定性，不能再制造新的事实源分散。
- 对 OpenSpec 的借鉴应停留在“产物 contract 更明确”这一层，而不是忽视 OpenHarness 已有的归档、验证和证据写回模型。
