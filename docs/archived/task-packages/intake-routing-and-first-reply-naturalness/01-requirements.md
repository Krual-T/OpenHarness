# Requirements

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## Goal
让 OpenHarness 在接收新请求时，能够先判断这次工作的真实主轴，再决定是否把 `openharness bootstrap` 作为前台入口动作，并把必须暴露的 workflow 信息转译成自然、克制的中文协作回复。

本轮的单一成功指标是：面对不同类型的仓库请求，agent 能稳定区分“需要 active task context 才能决定下一步”和“只需要把 `bootstrap` 当作后台上下文”的场景，且首轮用户可见回复不再主要由执行痕迹组成。

## Problem Statement
OpenHarness 现在已经把仓库入口、阶段路由、task package 协议和作者入口基本补齐，但实际首轮对话仍然暴露出一个新的摩擦：协议是清楚了，入口体验却不总是自然。

这次问题不是单纯的文案润色，而是 intake routing 与用户可见输出之间的边界没有收紧。当前容易出现几类错位：

- agent 过早把请求解释成“继续某个 active task package 的执行流”，于是很快前台运行 `openharness bootstrap`，即使用户真正要的是项目理解、整体评估、协议审查或局部讨论。
- 首轮回复同时混入“理解用户目标”“汇报当前阶段”“展示内部执行痕迹”“承诺下一步流程”，读起来像在回放状态机，而不是在与用户协作。
- `using-openharness` 要求显式说明当前阶段、刚完成什么、下一步做什么，但这些要求还没有被进一步约束成“用户应该看到什么”和“哪些内部动作不该直接暴露”。
- 历史任务已经解决“作者入口可发现”和“阶段可见性”，但没有明确处理“可见性”和“前台化”不是一回事，也没有给出 `bootstrap` 的适用边界。

如果不把这个问题单独收敛，OpenHarness 会继续出现一种结构性风险：仓库内部协议越来越完整，但用户第一轮看到的内容仍然像系统在自证合规，而不是先围绕用户请求建立共同工作面。

现在要做这件事，是因为真实使用已经暴露出这个摩擦，而且它已经不再是个别措辞问题。它关系到：

- `using-openharness` 是否会把所有仓库请求都拉向同一条入口路径；
- `bootstrap` 是否会从“上下文工具”被误用成“默认前台动作”；
- 用户是否能在首轮回复中看到真正相关的信息，而不是流程日志。

目标用户主要是：

- 首次与 OpenHarness 协作、对仓库协议还不熟的维护者；
- 需要用 OpenHarness 做项目评估、协议审查、局部讨论或活跃任务推进的协作者；
- 想判断这个仓库到底是“能协作的流程”还是“会劫持对话的协议”的观察者。

核心场景是：用户给出一个仓库相关请求后，agent 需要先识别这次请求的主轴，再决定是否需要 `bootstrap`、是否要进入某个 active task package，以及首轮回复应如何暴露 workflow 信息而不破坏自然度。

## Required Outcomes
1. 定义 intake routing 的任务分类边界。
   `acceptance criteria`：能够明确区分至少以下几类请求在入口时的不同处理方式：活跃任务推进、仓库整体评估/调研、协议或架构审查、局部代码或交互问题讨论、混合型请求。
2. 明确 `openharness bootstrap` 的使用角色，而不是只讨论它“能不能用”。
   `acceptance criteria`：写清楚什么时候它应作为前台入口动作，什么时候只能作为后台补充上下文，什么时候会把请求主轴带偏。
3. 给出首轮用户可见回复的最小约束。
   `acceptance criteria`：首轮回复仍然能交代当前阶段、刚完成的关键动作和下一步，但不直接把 `Explored`、`Ran ...`、原始路径输出或类似执行日志当作主内容。
4. 明确本轮设计应落在哪些协议表面。
   `acceptance criteria`：指出后续探索需要审查的文件和表面至少包括 `using-openharness`、相关 stage skill、CLI 行为说明、测试以及必要的任务文档，而不是停留在聊天感受层面。
5. 给出后续实现与验证的最小方向。
   `acceptance criteria`：后续设计阶段能够把需求映射成可执行改动，且验证场景至少覆盖“整体评估请求”“继续 active package 请求”“局部问题讨论请求”和“混合请求”。

## Non-Goals
- 本轮不简单把所有首轮回复都改成“不运行 `bootstrap`”。
- 本轮不否定阶段播报本身；要解决的是它如何被翻译成自然回复，而不是把阶段信息彻底隐藏。
- 本轮不直接把所有 skill 正文改写成中文文案工程。
- 本轮不重做整个 workflow status 体系，也不引入新的 task package 文件。
- 本轮不把 CLI 变成交互式问答系统。
- `counterexample`：如果只是把某一条具体回复润色得更顺，但没有定义什么时候该前台使用 `bootstrap`、什么时候不该，那不算完成本任务。

## Constraints
- 必须保留 `using-openharness` 作为仓库入口 skill 的地位，不能借解决自然度问题重新引入第二套入口系统。
- 必须尊重现有 task package 协议、状态流和作者入口结构；如果需要突破这些边界，应拆成后续 task package。
- 必须把“协议要求显式说明阶段”与“用户不需要看到原始执行日志”同时成立，而不是二选一。
- 若某项改动涉及 CLI、skill、文档和测试的联动，后续设计必须显式写出同步更新范围，避免再次出现协议与体验脱节。
- `cost cap`：本轮先聚焦 intake routing、首轮用户可见回复和 `bootstrap` 角色边界，不扩展到全仓库统一语气重写或所有历史对话模板清理。
