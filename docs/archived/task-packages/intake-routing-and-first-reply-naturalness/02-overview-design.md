# Overview Design

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## System Boundary
这轮设计覆盖的是 OpenHarness 在“收到用户请求后的第一轮 intake 与回复”这一段主路径，而不是重做整个 workflow 协议。

纳入范围的表面包括：

- `skills/using-openharness/SKILL.md` 中关于入口协议、阶段播报和 task-package 路由的规则。
- `skills/brainstorming/SKILL.md` 与 `skills/exploring-solution-space/SKILL.md` 中对 handoff 时机的表达方式。
- `openharness_cli/commands.py` 中 `bootstrap` 暴露给用户的角色与输出定位。
- 与入口行为相关的测试和协议文档。
- `OH-035` 自身的任务包文档，用来记录这轮关于 intake routing 与首轮回复自然度的设计结论。

不纳入范围的部分包括：

- task package 固定文件结构、状态流和归档协议。
- 将所有用户回复统一改写成单一风格模板。
- 重做 `bootstrap` 的核心列包能力或删除阶段可见性。
- 为所有 skill 新增完整中文翻译版本。
- 把对话层问题扩成新的 CLI 交互式问答系统。

## Proposed Structure
推荐采用“三层入口责任”的结构，而不是继续让一个动作同时承担上下文发现、主轴判断和用户沟通三件事。

1. intake routing 层负责判断当前请求的主轴
   - 落点主要在 `using-openharness` 及相关 stage skill 的入口协议。
   - 它先回答“这次请求到底是在推进 active task、理解整个仓库、审查协议、讨论局部问题，还是混合型请求”。
   - 只有当下一步确实依赖 active task context 时，`bootstrap` 才上升为前台入口动作。
2. context gathering 层负责补充仓库上下文
   - `bootstrap` 属于这一层的核心工具之一，但不是所有请求的前台主角。
   - 对评估、架构理解、局部讨论这类任务，`bootstrap` 可以后台运行，用来补活动任务、作者入口或状态流信息，但不主导首轮回复。
   - 对“帮我继续当前任务”“现在 active package 到哪一步了”这类请求，`bootstrap` 可以直接成为前台动作。
3. user-visible reply 层负责把协议信息翻译成自然协作语言
   - 首轮回复仍要交代当前阶段、刚确认的关键事实和下一步动作。
   - 但它不直接把 `Explored`、`Ran ...`、原始路径输出、CLI 原文或类似状态机日志拿来做主内容。
   - 这一层的目标不是“隐藏流程”，而是让流程信息服从用户当前请求。

主边界如下：

- `using-openharness` 负责决定要不要把 active task context 拉到台前。
- `bootstrap` 负责提供 task-package 和 stage 上下文，不负责替代主轴判断。
- stage skill 负责在进入对应阶段后说明“我现在在哪、刚完成什么、下一步做什么”，但这些说明应围绕用户任务组织，而不是围绕命令执行顺序组织。
- 测试负责把这些边界固定下来，避免后续再次把“可见性”扩张成“默认前台化”。

## Key Flows
主流程一：active task 推进请求

1. 用户明确在问当前活跃任务、下一阶段或继续推进现有 package。
2. intake routing 判断 active task context 是当前主轴。
3. agent 可以前台运行 `bootstrap`，并把结果转译成自然回复：当前处于什么阶段、刚确认了什么、下一步准备做什么。
4. 用户看到的是“与任务推进直接相关的阶段信息”，而不是完整 CLI 回放。

主流程二：仓库评估、协议审查或架构理解请求

1. 用户主要在问项目整体、协议合理性、架构取舍或体验问题。
2. intake routing 判断当前主轴不是 active task 推进。
3. agent 先说明自己会读哪些仓库表面，再视需要后台补 `bootstrap` 作为上下文来源之一。
4. 首轮回复以“我将如何建立评估框架 / 调查哪些表面”为主，不以前台列出 active package 或阶段信息为主。

主流程三：局部问题讨论或单点改动请求

1. 用户主要在问某个局部实现、某段回复、某个协议句子为什么奇怪。
2. intake routing 判断当前主轴是局部问题，而不是整个仓库流程。
3. agent 先读相关文件或证据；只有当局部问题直接依赖 active task 状态时，才补充 `bootstrap`。
4. 首轮回复聚焦当前问题本身，不把 task-package inventory 拉来抢主轴。

关键失败信号包括：

- 用户首轮看到的大部分内容仍然是命令、路径、`Explored` / `Ran` 之类执行痕迹。
- 明明是评估或局部讨论，请求却被过早转译成 active task 推进流。
- agent 虽然后台使用了 `bootstrap`，但没有向用户解释当前真正要解决的是什么。
- 另一种反向失败是：为了追求自然度，完全不再说明当前阶段、刚完成什么和下一步，导致协议信息消失。

## Stage Gates
- 必须明确 `bootstrap` 的三种角色边界：前台入口、后台上下文、暂不需要。
- 必须明确“主轴判断”先于“工具选择”，不能继续把 `bootstrap` 视为默认反射动作。
- 必须明确用户可见回复与内部执行痕迹之间的边界，避免再次出现日志主导首轮回复。
- 必须识别至少一个反向风险：为了自然而丢掉阶段播报与 workflow 透明度。
- 必须给出降级方向：如果暂时不改 CLI，也要先在 skill 协议和对话约束中固定边界；如果只改聊天约束不够稳，再补 CLI 或测试。

## Trade-offs
我比较了三种主要方案：

- 方案一：继续把 `bootstrap` 当作默认入口，只做文案润色
  - 收益是实现成本最低，现有协议和测试几乎不用动。
  - 代价是问题根因不变：错位来自入口判断，不只是回复措辞。
  - 不选原因：它只能让日志口吻变柔和，不能避免把不相关的 active task context 拉到台前。
- 方案二：把 `bootstrap` 从首轮流程中彻底移除
  - 收益是短期内最能避免协议感过重。
  - 代价是会损失现有的阶段可见性，也会让 active task 推进场景失去最自然的入口。
  - 不选原因：这会把“过度前台化”误治成“完全不用”，属于过度修正。
- 方案三：保留 `bootstrap`，但把它从默认动作改成条件化动作，并把首轮回复分层
  - 这是推荐方案。
  - 收益是既保留 task-package 协议的透明度，也避免把所有请求都拖入同一条执行路径。
  - 代价是需要同时更新 skill 协议、对话约束和部分测试，维护边界的成本更高。

降级方向：

- 如果本轮先不改 CLI，可以先把路由规则和用户可见回复边界写进 skill 与文档。
- 如果只靠 skill 文案仍然容易漂移，再补更明确的测试或 CLI 辅助信息。

## Overview Reflection
- 我先挑战了“这是不是纯文案问题”。结论是否定的。用户感到奇怪，不只是因为句子别扭，而是因为请求还没被正确分类，就已经触发了一个带强流程感的入口动作。
- 我也挑战了“是不是只要把 `bootstrap` 禁掉就行”。结论也是否定。`bootstrap` 在 active task 推进场景仍然是对的，问题在于它被当作默认反射动作，而不是条件化工具。
- 我进一步检查了“这是不是 CLI 问题”。结论是部分相关，但不是唯一根因。CLI 当前只是稳定地暴露阶段信息，真正缺的是上层协议没有明确规定这些信息什么时候该前台出现。
- 我接受的约束是：阶段信息必须保留，但要被转译成用户任务导向的自然中文。
- 我拒绝的方向是：把所有入口体验问题都推给单一文案模板或单一 CLI 改动。
- 我暂时延期的问题是：是否需要新增一套更正式的首轮回复测试夹具。触发条件是后续实现发现仅靠 skill 文档和现有测试仍不足以稳定约束行为。
