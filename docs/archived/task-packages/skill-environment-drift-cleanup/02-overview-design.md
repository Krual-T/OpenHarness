# Overview Design

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## System Boundary
- 覆盖范围是 `skills/` 下直接面向执行者的 skill 正文，以及少量会把错误能力继续传播出去的引用模板或示例文本。
- 重点边界是“当前仓库真实存在的协议与工具能力”和“skill 里写出来的默认假设”之间是否一致。
- 不纳入范围的是整个 skill 分层的大规模重构、运行时 helper 体系扩展，以及与本轮问题无关的风格润色。

## Proposed Structure
- 把环境真实性分成两层处理：
- 第一层是入口事实层，由 `AGENTS.md` 和 `using-openharness` 保持仓库级事实来源、任务包协议和验证要求。
- 第二层是执行辅助层，child skills 只保留与自身职责直接相关、且当前环境确实成立的动作说明。
- 对所有漂移点按三类动作收敛：
- 删除：当前环境不存在且无法由仓库协议支持的内容，例如 `TodoWrite`、`Task(...)`、固定 reviewer subagent type。
- 替换：目标行为仍然存在，但旧写法错误，例如把 Python 安装流程改为仓库统一的 `uv run ...` 约定。
- 降级：通用背景说明可以保留，但不能写成必须步骤，例如“可以选择使用隔离工作区”应与仓库实际限制一致。

## Key Flows
- 先根据 task package 列出受影响 skill 和漂移类型，明确哪些属于直接误导执行的高优先级问题。
- 修改时优先处理入口与执行链路上最容易被触发的 skill，例如 `using-git-worktrees` 和 `subagent-driven-development`。
- 每处修改都要能回答三个问题：当前仓库是否真的支持、是否仍属于该 skill 的职责、是否会与 `AGENTS.md` 或 `using-openharness` 冲突。
- 修改完成后用 harness 校验确认任务包协议仍然成立，再把执行证据写回任务包。

## Stage Gates
- 必须有一份明确的问题分类，至少覆盖旧入口约定、工具能力假设、命令约定冲突三类。
- 必须确认“仓库级事实”只由 `AGENTS.md` 和 `using-openharness` 兜底，child skills 不再自行发明环境前提。
- 必须给出失败模式与回退方向：如果某条说明既不能删除也无法证明为当前事实，就降级为非强制背景，不能继续写成硬规则。

## Trade-offs
- 收益是 skill 文档更可信，执行者不需要每次现查哪些词是旧仓库遗留物。
- 代价是需要直接修改多个 skill 文档，短期内会触碰一批跨文件文本，但这比继续容忍错误说明成本更低。
- 不选“仅补一个总说明、保留旧正文”的方案，因为执行者通常直接读被触发的 skill，旧内容依然会先误导人。
- 不选“顺手重构整个 skill 体系”的方案，因为这会把环境真实性问题和结构优化问题混成一轮，增加回归面。

## Overview Reflection
- 反思过的备选方向一是只修最明显的命令示例，二是一次性全面重写所有 skill。前者会遗漏能力假设层面的错配，后者超出本任务边界。
- 本轮接受的约束是先保证“不会误导当前仓库执行”，而不是追求所有 skill 长度都立刻最优。
- 主要风险是 OH-020 与 OH-021 会改到同一批文件，因此需要用职责边界来约束文本修改：本任务优先修正错误环境假设，结构去重只在不扩大范围时顺带收敛。
