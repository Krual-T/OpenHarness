# Overview Design

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## System Boundary
- 覆盖范围是 `using-openharness` 之外几个最常用 child skills 的协议文本边界，尤其是重复定义阶段制度、归档规则和写回职责的部分。
- 目标边界不是让 child skills 变成空壳，而是让它们只描述“本阶段该做什么”，把“仓库整体怎么运作”收敛到入口层。
- 不纳入范围的是改变 task package 状态流语义、另起新的入口 skill，或为每个阶段再新增一层共享文档。

## Proposed Structure
- 采用“一主多辅”的协议结构：
- `using-openharness` 保留仓库级常驻协议，包括事实来源、任务包顺序、阶段状态流、阶段门槛、challenge closure 与 archive 规则。
- `brainstorming`、`exploring-solution-space` 保留各自阶段的方法论、输出物和本阶段专属检查点，但不再重复整套仓库级制度。
- 执行型 helper skills 只保留各自操作约束，不再自带一套独立的任务编排或收尾制度。
- 当 child skill 需要依赖仓库级规则时，直接引用入口层而不是复制同义文本。

## Key Flows
- 协作者先经过 `using-openharness` 获得任务包协议与当前阶段。
- 进入某个 child skill 后，只补充该阶段需要的方法、输出物和注意事项。
- 当 child skill 需要说明“何时算阶段完成”时，只保留本阶段的局部门槛，仓库级状态推进依据入口层定义。
- 最后通过任务包与验证证据把阶段结果落回仓库，而不是让每个 child skill 都各自维护一套闭环描述。

## Stage Gates
- 必须明确列出哪些内容属于仓库级协议、哪些属于 child skill 专属动作。
- 必须给出至少一种替代方案并说明为什么不用，例如新增共享中间文档或继续多处重复维护。
- 必须定义失败模式：如果收敛后 child skill 无法独立指导其阶段动作，说明删得过头，需要回补阶段专属内容。

## Trade-offs
- 收益是减少重复修改面，仓库协议调整时只需要先改入口层，再少量修正 child skill 的阶段动作。
- 代价是 child skills 会更依赖读者接受“入口层是权威源”这一前提，不能再把所有规则都内嵌在自己体内。
- 不选“单独新建 protocol skill”的方案，因为这会重新制造第二个入口层，与当前仓库约束冲突。
- 不选“继续允许少量重复”的方案，因为这类重复已经在仓库里演化成分叉，维护成本是真实存在的。

## Overview Reflection
- 反思过的一个方向是把所有共享规则拆到新的 references 文档，再让 skill 统一引用。这个方向理论上更瘦，但会引入新的事实分散点，不适合当前仓库。
- 当前推荐方案把入口层继续做厚一点、child skills 做薄一点，代价最小，也最符合 `AGENTS.md` 已经规定的事实来源优先级。
- 主要验证风险是精简过度后 child skill 失去可执行性，因此实施时会优先保留阶段动作、删除重复制度说明。
