# Overview Design

> 章节标题保留英文；正文默认使用中文；命令、状态值、YAML 键名、文件名与路径保持英文。

## System Boundary
- 覆盖范围包括两类表面：活跃协议文档中的命令入口，以及验证这些说明是否一致的测试。
- 不纳入范围的部分是 `openharness_cli` 运行时行为、历史归档 task package、`update` 命令的安装升级路径，以及业务仓库本身的 Python 打包配置。

## Proposed Structure
- 文档层统一强调“标准入口是 `openharness <cmd>`”。
- skills 与 AGENTS 统一说明运行约束：“默认在项目根目录执行；若不在根目录，显式传 `--repo <项目根目录>`。”
- 兼容脚本入口继续保留，但不再出现在活跃协议文档中。

## Key Flows
1. 维护者阅读 AGENTS 或 `using-openharness` skill。
2. 文档只给出 `openharness <cmd>` 这一套入口。
3. 如果维护者不在项目根目录，文档要求他显式传 `--repo <项目根目录>`，而不是改走脚本路径命令。

## Stage Gates
- 需要明确技能文档不再出现脚本路径入口。
- 需要明确项目根目录约束与 `--repo` 的补救用法。
- 需要保持文档与当前 CLI 真实行为一致，不暗示自动发现根目录。

## Trade-offs
- 这次选择只改文档不改实现，是因为用户明确要求只在 skills 中强调，而不是扩张成 CLI 行为改造。
- 代价是用户仍需要理解当前 `--repo` 语义，但这是现有工具的真实行为，文档应当准确表达。
- 不选“继续保留脚本路径作为回退示例”，因为这会让标准入口再次分裂。

## Overview Reflection
- 原本考虑过把这轮做到“自动回溯仓库根目录”，但这已经超出用户当前要解决的问题。
- 用户当前要的是入口统一和认知约束统一，所以最终把方案收缩为文档层统一，不改变 CLI 语义。
