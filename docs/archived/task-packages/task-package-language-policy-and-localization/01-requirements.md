# Requirements

## Goal
为 OpenHarness 的 task package 制定并落地一套语言策略：优先保证维护者可以用中文顺畅阅读，同时保留仓库当前依赖的英文标识符、命令和校验表面。

## Problem Statement
仓库目前存在一个直接影响可维护性的缺口：

- 维护者并不希望频繁阅读长篇英文 task package 正文
- task package 是仓库里使用频率最高的事实载体之一
- 现有模板和校验规则仍然假定章节标题与正文都是英文优先

这会直接造成协议正确性与维护者可读性之间的冲突：

- task package 也许结构上仍然合法，但维护者阅读和审阅成本很高
- 仓库确实需要本地化（localization），但如果直接整包翻成中文，会破坏 `check-tasks`、测试和模板一致性
- 如果没有明确规则，文档会逐步漂移成不受控的中英混写状态

因此，这个项目必须对下面几个问题给出清晰答案：

- 哪些仓库表面应当改为中文优先
- 哪些表面因为受机器校验或协议稳定性约束而必须保留英文
- task package 的章节标题应该继续用英文、迁移成中文，还是同时支持两种写法
- 本地化应该怎样分阶段推进，才能不破坏当前仓库校验

## Required Outcomes
1. 明确 active 与 archived task package 的推荐语言策略，并把第一阶段规则真正落到仓库里。
2. 明确哪些结构化元素必须保持英文，包括状态值、YAML 键名、命令、文件名、路径和协议标识。
3. 给出可执行的分阶段本地化方案，区分“本轮立即落地”和“后续再做”的范围。
4. 明确章节标题本地化与 `check-tasks`、模板、测试之间的关系，并决定本轮不做中文标题迁移。
5. 更新会影响后续编写行为的仓库表面，让新建 task package 能直接遵守这套策略。

## Success Conditions
- 之后的维护者可以主要依靠中文正文阅读 task package，而不会失去协议一致性。
- 仓库对“中文正文、英文结构”和“全面本地化”之间的取舍有稳定答案。
- 后续本地化工作可以沿着明确路线推进，而不是继续零散修改。
- 仓库协议明确承认：维护者可读性是产品需求，而不是个人偏好。

## Non-Goals
- 本轮不翻译整个仓库的所有文件。
- 本轮不本地化代码、CLI 命令、YAML 键名或状态值。
- 本轮不把所有 skill 文件整体翻译成中文。
- 本轮不改动校验器去支持中文章节标题。

## Constraints
- 当前 `check-tasks` 会依赖若干英文章节标题进行校验。
- 命令、状态名、YAML 键名和其他协议标识在没有专门实现包之前必须保持稳定。
- 方案必须在提升阅读体验的同时，避免制造第二套平行文档协议。
- 语言策略必须明确区分“给人看的叙述正文”和“对机器敏感的结构锚点”。
